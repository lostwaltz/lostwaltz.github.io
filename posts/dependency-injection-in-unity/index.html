유니티(Unity) 의존성 주입(DI)으로 싱글톤의 한계 극복하기

2026.03.06 · 7분 소요 unity architecture devlog di

최근 새로운 데스크탑 에이전트 프로젝트를 진행하면서, 기존에 상용화된 유사 오픈소스 프로젝트의 코드를 뜯어볼 기회가 있었다. 기능적으로는 훌륭했지만, 내부 코드는 수십 개의 매니저 클래스가 얽히고설킨 거대한 스파게티 구조였다.

원인은 명확했다. 싱글톤(Singleton) 패턴의 남용과 유니티 라이프사이클에 대한 강한 결합 때문이었다. 이를 반면교사 삼아, 현재 진행 중인 프로젝트에 의존성 주입(Dependency Injection, DI)을 도입하게 된 배경과 그 구조적 이점을 정리해 본다.

기존 구조의 문제 (싱글톤 남용)

유니티 생태계에서 싱글톤은 마약과도 같다. PlayerManager.Instance.GetPlayer() 한 줄이면 씬(Scene) 어디서든 원하는 데이터에 접근할 수 있기 때문이다. 하지만 프로젝트 규모가 커지면 다음과 같은 치명적인 문제들이 발생한다.

  • 강한 결합도 (Tight Coupling): 특정 객체를 단독으로 테스트하고 싶어도 엮여있는 수많은 Manager들이 없으면 NullReferenceException을 뿜으며 뻗어버린다.
  • 실행 순서(Lifecycle)의 카오스: 어떤 매니저의 Awake()가 먼저 실행될지 보장할 수 없기 때문에, 초기화 타이밍이 꼬이는 이른바 '기도 메타' 프로그래밍이 된다.
  • 기능 확장(Modding)의 불가: 새로운 기능을 추가하려면 기존 매니저 클래스 내부의 if-else 문을 직접 수정해야만 한다. (OCP 위배)
// ❌ 기존 방식 (싱글톤 남용)
public class Monster : MonoBehaviour 
{
    void Update() 
    {
        // 몬스터가 PlayerManager의 존재를 명시적으로 알아야만 함
        // PlayerManager가 씬에 없으면 당장 에러 발생
        Vector3 target = PlayerManager.Instance.GetPlayer().position;
        MoveTo(target);
    }
}

의존성 주입(DI) 패턴으로 전환

이러한 한계를 극복하기 위해 "객체가 자신이 사용할 도구를 직접 찾지 않고, 생성 시점에 외부에서 주입받는" DI 패턴을 도입했다. 핵심은 "나는 네가 누군지 알 필요 없다"는 철학이다.

1. 인터페이스(Interface)를 통한 추상화

객체들은 구체적인 클래스 대신 인터페이스에만 의존하게 만든다.

// ✅ 새로운 방식 (DI 적용)
public class Monster : MonoBehaviour 
{
    private IPlayer _targetPlayer; // 인터페이스에만 의존

    // 팩토리나 Bootstrapper가 생성 시점에 타겟을 "주입"해줌
    public void Construct(IPlayer target) 
    {
        _targetPlayer = target;
    }

    void Update() 
    {
        if (_targetPlayer == null) return;
        // 타겟이 진짜 유저인지, 분신인지, 테스트용 더미인지 알 필요 없음
        MoveTo(_targetPlayer.Position); 
    }
}

2. 단일 진입점 (AppBootstrapper)

의존성들이 꼬이지 않도록, 시스템 최상단에 모든 조립을 통제하는 단일 진입점(AppBootstrapper)을 둔다. 유니티의 Awake/Start에 흩어져 있던 초기화 로직을 한 곳으로 모으는 것이다.

public class AppBootstrapper : MonoBehaviour 
{
    void Awake() 
    {
        // 1. 핵심 인프라 생성
        IEventBus eventBus = new EventBus();
        ILLMProvider llmProvider = new GroqProvider();
        
        // 2. 핵심 로직 조립 (순수 C# 지향)
        AgentLoop agent = new AgentLoop(eventBus, llmProvider);
        
        // 3. UI 및 뷰 계층에 의존성 주입
        FindObjectOfType<UIManager>().Construct(eventBus);
    }
}

모딩(Modding)과 확장성의 마법

DI와 EventBus가 결합되면 프로젝트는 놀라운 확장성을 갖게 된다. 유명한 모딩 친화적 게임(예: 마인크래프트)들이 플러그인을 지원하는 원리도 이와 같다. 메인 로직(AgentLoop)은 ITool이라는 인터페이스 가방(ToolRegistry)만 들고 있을 뿐, 그 안에 무엇이 들어있는지 신경 쓰지 않는다.

만약 누군가 "디스코드 알림 기능"을 추가하고 싶다면? 기존 코드를 단 한 줄도 건드릴 필요 없이, ITool을 상속받는 DiscordTool 클래스를 만들어 AppBootstrapper에서 레지스트리에 찔러 넣어주기만 하면 끝난다.

정리

당장 눈앞의 기능을 구현하기엔 전역 매니저(Singleton)를 사용하는 것이 압도적으로 빠르다. 하지만 장기적인 유지보수와 진정한 의미의 로직 모딩(Logic Modding)을 고려한다면, 느려 보이더라도 DI 기반의 아키텍처를 뼈대로 잡는 것이 결국 가장 빠른 길이다.

구분 기존 구조 (Singleton) DI + EventBus 아키텍처 (Proposed)
진입점 각 MonoBehaviour의 Awake/Start AppBootstrapper로 통제 (단일 진입)
결합도 매니저 클래스 간 강한 결합 (Tight) 인터페이스 기반 느슨한 결합 (Loose)
테스트 단독 테스트 사실상 불가 Mock 주입을 통한 완벽한 단독 테스트 가능
기능 추가 기존 클래스 코드를 직접 수정해야 함 외부 플러그인(모드) 형태로 주입만 하면 됨
