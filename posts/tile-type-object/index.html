Unity 타일 기믹 시스템 재설계
— Type Object 패턴

2026.03.01 · 8분 unity architecture devlog

개발 중인 게임 Rolice는 주사위를 굴려 타일 색상과 바닥면 색상을 맞추는 퍼즐 게임이다. 핵심 재미가 타일 기믹에서 나오는 만큼, 타일 시스템은 확장성이 중요했다. 그런데 기믹을 하나씩 추가하다 보니 구조가 점점 엉망이 되어갔다.

기존 구조의 문제

초기 설계는 타일 기믹마다 RcTileBehaviorSO를 상속한 클래스를 하나씩 만드는 방식이었다. 색상 매칭 타일은 RcColorMatchBehaviorSO, 돌 타일은 RcStoneBehaviorSO처럼. 

// 기존 방식 — 기믹마다 Behavior 클래스 작성
public class RcColorMatchBehaviorSO : RcTileBehaviorSO
{
    public override ITileBehavior CreateBehavior(GameObject obj, RcTileData data)
    {
        return new RcColorMatchBehavior(this, obj, data);
    }
}

public class RcColorMatchBehavior : ITileBehavior
{
    public void OnEnter(RcDicePawn pawn)
    {
        if (pawn.GetBottomColor() == tileData.Color)
            ClearTile(); // 조건 체크 + 동작 + 상태 관리가 한 클래스에 몰림
    }
}

당장은 괜찮아 보였지만 문제가 쌓이기 시작했다.

  • 조합 불가 — "색상 매칭 + 돌 내구도"를 동시에 가진 타일을 만들려면 새 클래스를 또 작성해야 했다.
  • 관심사 혼재 — 조건 체크, 시각 피드백, 상태 관리가 한 클래스에 뒤섞였다.
  • 재사용 불가 — 비슷한 로직이 여러 Behavior에 복붙되기 시작했다.

타일 기믹 종류가 늘어날수록 클래스 수가 선형으로 늘어나는 구조였다. 이건 고쳐야 했다.

Type Object 패턴으로 전환

핵심 아이디어는 "타일 타입을 코드가 아닌 데이터로 정의"하는 것이다. 각 타일 종류를 RcTileTypeSO라는 ScriptableObject로 표현하고, 그 안에 어떤 규칙을 가질지를 선언적으로 구성한다. 

// Type Object — 타일 타입 정의 (데이터)
[CreateAssetMenu(menuName = "Rolice/Tile/Type")]
public class RcTileTypeSO : ScriptableObject
{
    public GameObject Prefab;             // 비주얼
    public RcTileRuleSO[] Rules;          // 적용할 규칙 목록
    public bool RequiresClearTracking;    // 클리어 추적 여부
}

RcTileTypeSO는 행동 로직을 직접 갖지 않는다. 어떤 Rules를 적용할지만 선언한다. 동작의 책임은 RcTileRuleSO로 위임된다.

Rule = Condition + Action 조합

각 Rule은 조건(Condition)과 동작(Action)의 조합이다. 조건이 모두 통과되면 동작을 실행한다. 조건과 동작은 각각 독립된 ScriptableObject로 분리되어 있어 재사용이 자유롭다. 

// Rule — 조건이 통과되면 동작 실행
[CreateAssetMenu(menuName = "Rolice/Tile/Rule")]
public class RcTileRuleSO : ScriptableObject
{
    public RcTileConditionSO[] Conditions;
    public RcTileActionSO[]    Actions;

    public bool TryExecute(RcDicePawn pawn, RcTileData tileData)
    {
        foreach (var cond in Conditions)
            if (!cond.Evaluate(pawn, tileData)) return false;

        foreach (var action in Actions)
            action.Execute(pawn, tileData);

        return true;
    }
}

// 조건 베이스
public abstract class RcTileConditionSO : ScriptableObject
{
    public abstract bool Evaluate(RcDicePawn pawn, RcTileData tileData);
}

// 동작 베이스
public abstract class RcTileActionSO : ScriptableObject
{
    public abstract void Execute(RcDicePawn pawn, RcTileData tileData);
}

구체적인 구현 예시

// 조건 — 주사위 바닥면 색상이 타일 색상과 일치하는가
public class RcColorMatchConditionSO : RcTileConditionSO
{
    public override bool Evaluate(RcDicePawn pawn, RcTileData tileData)
    {
        if (tileData is not RcColorTileData colorTile) return false;
        return pawn.GetBottomColor() == colorTile.Color;
    }
}

// 동작 — 타일 클리어 처리
public class RcClearTileActionSO : RcTileActionSO
{
    public Material ClearedMaterial;
    public GameObject ClearEffectPrefab;

    public override void Execute(RcDicePawn pawn, RcTileData tileData)
    {
        if (tileData is not RcColorTileData colorTile || colorTile.IsCleared) return;

        colorTile.IsCleared = true;
        ApplyMaterial(tileData.TileObject, ClearedMaterial);
        SpawnEffect(tileData.TileObject.transform.position);
        RcLevelManager.Instance.ClearColorTile(gridPos);
    }
}

다형적 타일 데이터 계층

타일 기믹마다 필요한 런타임 상태가 다르다. 색상 타일은 IsCleared가 필요하고, 돌 타일은 HitCount가 추가로 필요하다. 이를 상속으로 깔끔하게 분리했다. 

// 기본 타일 데이터
public class RcTileData
{
    public RcTileTypeSO TileType;
    public bool bCanEnter = true;

    public virtual RcTileData Clone() => new RcTileData { TileType = TileType, bCanEnter = bCanEnter };
}

// 색상 타일 — IsCleared 추가
public class RcColorTileData : RcTileData
{
    public RcColorSO Color;
    [NonSerialized] public bool IsCleared;

    public override RcTileData Clone() =>
        new RcColorTileData { TileType = TileType, bCanEnter = bCanEnter, Color = Color };
}

// 돌 타일 — HitCount 추가
public class RcStoneTileData : RcColorTileData
{
    [NonSerialized] public int HitCount;

    public override RcTileData Clone() =>
        new RcStoneTileData { TileType = TileType, bCanEnter = bCanEnter, Color = Color };
}

Condition과 Action에서 is not RcColorTileData 패턴 매칭으로 타입 안전하게 분기한다. 잘못된 타입의 타일에 규칙이 적용되는 일이 없다.

새 기믹 추가하는 법

재설계 후 새 타일 기믹을 추가하는 과정이 크게 단순해졌다.

기존: 새 기믹 = 새 Behavior 클래스 작성 → 빌드 필요
변경 후: 기존 Condition/Action SO를 조합하거나, 새 Action SO 하나만 추가

예를 들어 "색상 매칭이 되어야 클리어되는 돌 타일"을 만들고 싶다면:

StoneTileType (RcTileTypeSO)
├── Rule A: [ColorMatchCondition] + [HitCountUnder3Condition] → [BreakStoneAction]
└── Rule B: [ColorMatchCondition] + [StoneDepletedCondition]  → [ClearTileAction]

코드를 한 줄도 건드리지 않고 에디터에서 SO를 조합하는 것만으로 완성된다. 기획 변경에도 유연하게 대응할 수 있다.

정리

항목 기존 변경 후
기믹 정의 Behavior 클래스 작성 RcTileTypeSO에 Rule 조합
새 기믹 추가 새 클래스 + 빌드 SO 에셋 생성만으로 완료
기믹 조합 전용 클래스 별도 작성 Rule 여러 개 연결
타일 상태 단일 RcTileData 다형적 계층 (RcColorTileData 등)
재사용성 낮음 Condition/Action 자유롭게 공유

Type Object 패턴이 Unity ScriptableObject와 잘 어울린다는 걸 이번에 체감했다. 런타임 데이터와 설계 데이터를 분리하고, 행동을 조건과 동작의 조합으로 표현하는 구조가 기획 변경이 잦은 게임 개발에서 특히 유용하다.

아직 Rolice는 개발 중이다. 기믹이 더 추가될수록 이 구조가 얼마나 버텨주는지 두고 볼 생각이다.
