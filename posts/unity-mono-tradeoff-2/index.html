그래서 어떻게 짰나
— 절충안 구현 흐름

2026.03.09 · 6분 unity architecture devlog

이전 글에서 Mono, SO, 순수 C# 각각의 트레이드오프를 정리했다. 결론은 "완전한 해결책은 없다"였는데, 그렇다고 손 놓고 있을 수는 없다. 이 글은 그 세 가지를 조합해 사이드이펙트를 최소화한 절충안을 실제 코드로 정리한다.

구조 개요

SO 팩토리 역할 — 프리팹, CreateBehavior()
Params 인스턴스별 수치 — [SerializeReference] 직렬화
Behavior 순수 C# 동작 — 상태 보유, Mono 없음
TileSystem 중앙 관리 — Dictionary로 등록/디스패치

각 레이어가 하나의 책임만 갖는다. SO는 타입을 정의하고, Params는 수치를 들고, Behavior는 동작을 실행하고, TileSystem은 연결한다.

1. 데이터 — TileData + SerializeReference Params

타일 데이터는 두 가지로 나뉜다. 타입이 뭔지(TileType)와 이 타일 인스턴스의 수치가 뭔지(Params). 

[Serializable]
public class RcTileData
{
    public RcTileTypeSO TileType;
    [SerializeReference] public RcTileParams Params;

    public bool IsEmpty => TileType == null;
}

[Serializable] public abstract class RcTileParams { }

[Serializable]
public class RcColorTileParams : RcTileParams
{
    public RcColorType Color;
}

[Serializable]
public class RcStoneTileParams : RcTileParams
{
    public RcColorType Color;  // 깎인 뒤 드러나는 색상
    public int MaxHits = 1;
}

[SerializeReference]를 쓰면 인스펙터에서 Params 타입을 드롭다운으로 선택할 수 있다. RcTileData에 필드가 늘어나지 않고, 기믹마다 필요한 수치를 독립된 클래스에서 관리한다.

수평 상속을 유지하는 게 중요하다. RcStoneTileParams가 RcColorTileParams를 상속하면 Stone이 Color에 결합되어, Color가 바뀔 때 Stone도 영향을 받는다. 공통 필드가 중복되더라도 수평으로 유지하는 편이 낫다.

2. SO — 팩토리

RcTileTypeSO는 이 타입의 프리팹과 CreateBehavior()만 갖는다. 공유 설정(머티리얼, 기본 사운드 등)을 여기에 두면 된다. 

public abstract class RcTileTypeSO : ScriptableObject
{
    public GameObject Prefab;
    public abstract RcTileBehavior CreateBehavior();
}

public class RcColorTileTypeSO : RcTileTypeSO
{
    public override RcTileBehavior CreateBehavior() => new RcColorTileBehavior();
}

public class RcStoneTileTypeSO : RcTileTypeSO
{
    public override RcTileBehavior CreateBehavior() => new RcStoneTileBehavior();
}

새 기믹을 추가할 때 RcTileTypeSO 서브클래스 하나와 SO 에셋 하나만 만들면 된다. 기존 코드는 건드리지 않는다.

3. Behavior — 순수 C# 동작

Mono가 아닌 순수 C# 클래스다. GameObject도 없고 Update도 없다. 동작에 필요한 상태(_hits 등)는 인스턴스 필드로 들고 있는다. 

public abstract class RcTileBehavior
{
    public abstract void OnDiceEnter(RcTileData data, RcDicePawn pawn, Vector2Int pos);
    public virtual void Reset() { }  // 레벨 리셋 시 상태 초기화
}

public class RcColorTileBehavior : RcTileBehavior
{
    private bool _cleared;

    public override void OnDiceEnter(RcTileData data, RcDicePawn pawn, Vector2Int pos)
    {
        if (_cleared) return;
        if (data.Params is not RcColorTileParams p) return;

        if (pawn.GetBottomColor() == p.Color)
        {
            _cleared = true;
            RcTileSystem.Instance.ClearTile(pos);
        }
    }

    public override void Reset() => _cleared = false;
}

public class RcStoneTileBehavior : RcTileBehavior
{
    private int _hits;

    public override void OnDiceEnter(RcTileData data, RcDicePawn pawn, Vector2Int pos)
    {
        if (data.Params is not RcStoneTileParams p) return;

        _hits++;
        if (_hits >= p.MaxHits)
            RcTileSystem.Instance.ConvertToColor(pos, p.Color);
    }

    public override void Reset() => _hits = 0;
}

RcStoneTileBehavior는 내부 상태(_hits)만 바꾸고 RcTileSystem.ConvertToColor()를 호출한다. 런타임에 컴포넌트를 교체하지 않는다.

4. TileSystem — 중앙 관리

레벨이 빌드될 때 모든 타일을 등록받고, 다이스 이동이 완료될 때 해당 위치의 Behavior를 디스패치한다. 

public class RcTileSystem : RcSingletonMono<RcTileSystem>
{
    private Dictionary<Vector2Int, RcTileBehavior> _behaviors = new();
    private Dictionary<Vector2Int, RcTileData>     _tileMap   = new();

    public void Register(Vector2Int pos, RcTileData data, RcTileBehavior behavior)
    {
        _tileMap[pos]   = data;
        _behaviors[pos] = behavior;
    }

    public void OnDiceEnter(Vector2Int pos, RcDicePawn pawn)
    {
        if (_behaviors.TryGetValue(pos, out var behavior))
            behavior.OnDiceEnter(_tileMap[pos], pawn, pos);
    }

    public void ClearTile(Vector2Int pos) { /* 클리어 이벤트 발행 */ }

    public void ConvertToColor(Vector2Int pos, RcColorType color)
    {
        // Behavior를 ColorTileBehavior로 교체 (런타임 컴포넌트 교체 아님)
        var newData = new RcTileData
        {
            TileType = _tileMap[pos].TileType,
            Params   = new RcColorTileParams { Color = color }
        };
        _tileMap[pos]   = newData;
        _behaviors[pos] = new RcColorTileBehavior();
        // 비주얼 갱신 이벤트 발행
    }

    public void ResetAll()
    {
        foreach (var b in _behaviors.Values) b.Reset();
    }
}

5. 빌드 흐름

public class RcLevelManager
{
    public void BuildLevel(RcLevelDataSO levelData)
    {
        foreach (var (tileData, gridPos) in levelData.Tiles)
        {
            if (tileData.IsEmpty) continue;

            // 비주얼 — GameObject (Mono 없음)
            Instantiate(tileData.TileType.Prefab, GridToWorld(gridPos), Quaternion.identity);

            // 동작 — 순수 C# 인스턴스
            var behavior = tileData.TileType.CreateBehavior();
            RcTileSystem.Instance.Register(gridPos, tileData, behavior);
        }
    }
}

타일 GameObject에는 아무 스크립트도 붙지 않는다. 비주얼과 로직이 완전히 분리되어 있어서, 비주얼만 교체하거나 로직만 수정하는 게 독립적으로 가능하다.

남아있는 비용

이 구조도 공짜는 아니다.

  • [SerializeReference] 기본 인스펙터 UX가 불편 — 커스텀 PropertyDrawer가 필요하다
  • Params is not RcColorTileParams 패턴 매칭이 각 Behavior마다 반복된다
  • 레벨 리셋 시 ResetAll()을 명시적으로 호출해야 한다
  • ConvertToColor 같은 타일 전환 로직이 TileSystem에 집중되어 그 클래스가 커질 수 있다

다만 이 비용들은 관리 가능한 범위다. 런타임 컴포넌트 교체나 Mono 100개보다 훨씬 예측 가능하고 통제하기 쉽다.

유니티에서 "완전한 구조"는 없지만, 어떤 비용을 의식하고 선택했는지는 다르다. 이 구조의 핵심은 각 레이어가 하나의 책임만 갖도록 경계를 명확히 긋는 것이다. 그 경계가 흐릿해지는 순간 다시 처음으로 돌아간다.
