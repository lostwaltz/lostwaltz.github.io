Unity에서 완전한 설계란 없다
— Mono, SO, 순수 C# 트릴레마

2026.03.08 · 7분 unity architecture devlog

협업자가 올린 커밋 하나를 보다가 오래된 물음이 다시 떠올랐다. 타일 시스템 리팩터링이었는데, 기존의 복잡한 ScriptableObject 조합 구조를 걷어내고 MonoBehaviour 직접 상속으로 교체한 코드였다. 코드는 분명히 읽기 쉬워졌다. 그런데 뭔가 찜찜했다.

타일에 MonoBehaviour를 붙인다는 건, 타일이 100개면 컴포넌트가 100개라는 뜻이다. 내가 원래 일부러 피했던 바로 그 구조였다.

세 가지 선택지와 각각의 대가

유니티에서 "객체가 동작을 갖게 하는" 방법은 크게 세 가지다. 그리고 어느 것도 공짜가 아니다.

MonoBehaviour 컴포넌트

씬과 자연스럽게 연동되고, 라이프사이클(Awake, OnEnable, Update)을 자동으로 얻는다. 인스펙터에서 바로 확인할 수 있고, 인스턴스마다 독립적인 상태를 갖기도 쉽다.

대신 컴포넌트 수가 선형으로 늘어난다. 타일 100개면 MonoBehaviour 100개. Update를 안 쓰더라도 유니티 내부에서 관리 비용이 발생하고, 런타임에 AddComponent / Destroy로 컴포넌트를 교체하는 패턴은 버그의 온상이 된다. 

// 돌 타일이 깎이면 색상 타일로 "변환"
private void ConvertToColorTile()
{
    var colorTile = gameObject.AddComponent<RcColorTile>();
    colorTile.Construct(new RcTileData { Color = targetColor });
    Destroy(this); // 프레임 끝에 제거 — 그 사이 두 컴포넌트가 공존
}

Destroy는 즉시 실행되지 않는다. 한 프레임 동안 두 컴포넌트가 동시에 존재하고, 외부에서 캐싱하고 있던 참조는 null이 된다. 잡기 어려운 버그가 여기서 생긴다.

ScriptableObject 동작

SO는 에셋으로 존재하기 때문에 공유가 쉽고 인스펙터 지원도 받을 수 있다. 런타임에 인스턴스를 새로 만들지 않아도 된다.

문제는 SO가 인스턴스 상태를 가질 수 없다는 것이다. SO는 에셋이라 모든 타일이 같은 SO를 공유하기 때문에, 타일별로 달라지는 상태 (돌 타일의 남은 내구도, 색상 타일의 클리어 여부 등)는 별도 데이터 클래스에 들어가야 한다. 동작이 세분화될수록 SO 에셋 수가 폭발하고, 조합이 복잡해질수록 에디터가 지저분해진다.

순수 C# 클래스

Mono도 SO도 아닌, 그냥 C# 클래스 계층으로 동작을 정의하는 방법이다. 오버헤드가 없고 OOP를 온전히 활용할 수 있다.

대신 유니티와의 연결이 끊긴다. 인스펙터에서 보이지 않고, 씬 라이프사이클도 없다. 누가 이 인스턴스를 만들고, 들고 있고, 해제할지를 전부 직접 관리해야 한다. 관리 로직이 한 곳에 모이지 않으면 금방 흩어진다.

조합하면 해결될까

세 방식을 섞으면 어떨까. 팩토리는 SO로, 동작 인스턴스는 순수 C# 클래스로, 진입점만 Mono로 최소화하는 구조다. 

// SO는 팩토리 역할만
public abstract class RcTileTypeSO : ScriptableObject
{
    public abstract RcTileBehavior CreateBehavior();
}

// 동작은 순수 C# 계층
public abstract class RcTileBehavior
{
    public abstract void OnDiceEnter(RcTileData data, RcDicePawn pawn, Vector2Int pos);
}

// 중앙 시스템이 인스턴스 관리
public class RcTileSystem : RcSingletonMono<RcTileSystem>
{
    private Dictionary<Vector2Int, RcTileBehavior> _behaviors;

    public void OnDiceEnter(Vector2Int pos, RcDicePawn pawn)
    {
        if (_behaviors.TryGetValue(pos, out var behavior))
            behavior.OnDiceEnter(_tileMap[pos], pawn, pos);
    }
}

이렇게 하면 타일에 MonoBehaviour가 붙지 않고, SO 남발도 없고, 동작은 상속으로 확장 가능하다. 그나마 균형에 가까운 구조다.

하지만 이것도 공짜가 아니다. 타일별 커스텀 파라미터를 인스펙터에서 조정하기 어렵고, 인스턴스 관리 책임이 RcTileSystem에 집중되어 그 클래스가 복잡해진다. 팀 컨벤션이 없으면 누군가 다시 Mono를 붙이기 시작한다.

이게 유니티의 구조적 한계다

어떤 선택을 해도 사이드이펙트가 생기는 건 실력 부족이 아니다. 유니티가 MonoBehaviour 기반으로 설계된 엔진이기 때문에, Mono를 피하려고 할수록 엔진과 싸우는 꼴이 된다.

유니티가 DOTS(ECS)를 만든 건 이 구조적 문제를 스스로 인정한 것이다.
Mono 기반 설계의 한계를 엔진 레벨에서 해결하려 한 시도. 다만 러닝커브가 너무 높아 중소 프로젝트에선 현실적으로 쓰기 어렵다.

이번 협업 충돌도 그 연장선이었다. 협업자는 복잡한 SO 조합을 걷어내고 싶었고, 나는 타일마다 Mono가 붙는 걸 피하고 싶었다. 둘 다 맞는 판단이고, 둘 다 비용이 있었다.

그래서 설계란

"어떻게 하면 사이드이펙트가 없을까"가 아니라 "어떤 사이드이펙트를 감수할 것인가"를 결정하는 일이다.

방식 얻는 것 잃는 것
MonoBehaviour 인스턴스 관리 자동, 인스펙터 연동 컴포넌트 비용, 런타임 교체 위험
ScriptableObject 에셋 공유, 에디터 지원 인스턴스 상태 불가, SO 남발
순수 C# 오버헤드 없음, OOP 자유 인스펙터 단절, 라이프사이클 직접 관리

프로젝트 규모, 팀 구성, 기획 변경 빈도에 따라 어떤 비용을 선택할지가 달라진다. 그 판단 기준을 팀이 명시적으로 합의해두지 않으면, 코드리뷰마다 같은 논쟁이 반복된다. 이번 충돌이 정확히 그 케이스였다.

완전한 해결책을 찾는 것보다, 지금 선택의 트레이드오프를 팀이 함께 이해하고 있는 게 더 중요하다.

그래도 "그나마 나은 절충안"은 있다. 다음 글에서 SO 팩토리 + 순수 C# 동작 계층 + [SerializeReference] 파라미터를 조합한 실제 구현 흐름을 정리했다.
